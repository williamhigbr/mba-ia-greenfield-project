### pg-boss

**Context7:** `/timgit/pg-boss` · **Version:** `^10.x` · Used by TD-01 (queue) + TD-08 (status/failure lifecycle).

PostgreSQL-backed job queue. Single instance object shared by producer (API) and consumer (worker); the worker runs in a separate process (TD-04) that constructs its own `PgBoss` against the same database.

**Lifecycle — construct + start (both API producer and worker):**

```js
const { PgBoss } = require('pg-boss');
const boss = new PgBoss('postgres://user:pass@host/database'); // or { connectionString, schema }
boss.on('error', console.error);       // always attach — errors are emitted, not thrown
await boss.start();                    // creates/migrates the `pgboss` schema on first run
await boss.createQueue(queue);         // idempotent; declare each queue once at boot
```

- `boss.start()` bootstraps pg-boss's own schema (default schema name `pgboss`) — **independent of the app's TypeORM entities/migrations**; do not model pg-boss tables as TypeORM entities.
- Connection: pass a connection string OR an options object (`{ connectionString, schema, ... }`). Reuse the same `databaseConfig` inputs as the app (Docker service name `db` as host, per conventions).

**Producer — enqueue a processing job (API, on upload complete):**

```js
const jobId = await boss.send('video-process', { videoId }, {
  retryLimit: 3,          // default 2
  retryDelay: 30,         // seconds; default 0
  retryBackoff: true,     // exponential backoff on retryDelay
  retryDelayMax: 600,     // cap when retryBackoff is on
});
// send() resolves the job id (or null under throttling/unique-job opt-ins)
```

- `send(name, data, options)` — `data` is the JSONB payload; keep it minimal (`{ videoId }`) and let the worker load the row.
- Use the `id` option (a uuid) or `singletonKey` if you need idempotent enqueue (avoid double-processing the same video).

**Consumer — worker handler (separate worker process, TD-04):**

```js
await boss.work('video-process', async ([ job ]) => {
  // idempotent, keyed by job.data.videoId (TD-08): safe to re-run on retry
  await processVideo(job.data.videoId);
});
```

- `work(name, options, handler)` — default polls 1 job / 2s; batch via `batchSize`. Returning resolves the job `completed`; **throwing** fails it → pg-boss retries per `retryLimit`/`retryBackoff`, then routes to the dead-letter queue.
- Low-latency: create the queue with `notify: true` to get `LISTEN/NOTIFY` wake-ups (polling becomes a slow backstop).

**Retry + dead-letter (maps to TD-08 failure lifecycle):**

- Configure per-queue on `createQueue(name, { retryLimit, retryDelay, retryBackoff, retryDelayMax, deadLetter: 'video-process-dlq' })` OR per-job via `send()` options.
- `deadLetter` — name of the queue that receives jobs after all retries are exhausted. Pattern: create a `*-dlq` queue; a listener on it sets the video row to `failed` with an error reason (TD-08 terminal state).
- Terminal failure without a DLQ = job simply fails after retries.

### @aws-sdk/client-s3

**Context7:** `/aws/aws-sdk-js-v3` · **Version:** `^3.x` · Used by TD-02 (multipart upload) + TD-03 (object ops / key layout).

S3 client, S3-compatible with MinIO. Configure the client once (server-only storage module) with a custom endpoint + path-style addressing for MinIO:

```ts
import { S3Client } from "@aws-sdk/client-s3";
const s3 = new S3Client({
  region: "us-east-1",              // arbitrary but required
  endpoint: env.S3_ENDPOINT,        // e.g. http://minio:9000 (Docker service name)
  forcePathStyle: true,             // REQUIRED for MinIO (path-style, not vhost-style)
  credentials: { accessKeyId: env.S3_ACCESS_KEY, secretAccessKey: env.S3_SECRET_KEY },
});
```

**Multipart upload handshake (TD-02) — the API orchestrates initiate/complete; parts are PUT directly by the browser via presigned URLs:**

- `CreateMultipartUploadCommand` → returns an `UploadId`. Input: `{ Bucket, Key }` (Key = `videos/{videoId}/original.<ext>` per TD-03).
- `UploadPartCommand` (PUT `/{Bucket}/{Key}?partNumber&uploadId`) → one per part; parts ≥5MB except the last. The browser uploads each part to a **presigned** URL (see s3-request-presigner) — bytes never transit the API.
- `CompleteMultipartUploadCommand` → assembles parts: input `{ Bucket, Key, UploadId, MultipartUpload: { Parts: [{ ETag, PartNumber }, ...] } }`. Client returns each part's `ETag`; the API collects them (or the browser posts them back) to complete.
- Reap incomplete uploads with a bucket lifecycle rule (AbortIncompleteMultipartUpload) — TD-02/TD-03.

**Object ops (TD-03):** `PutObjectCommand` / `GetObjectCommand` / `HeadObjectCommand` / `DeleteObjectCommand` — keyed by the deterministic prefixes (`videos/{videoId}/…`, `thumbnails/{videoId}/…`). Persist the **key**, never a full URL.

### @aws-sdk/s3-request-presigner

**Context7:** `/aws/aws-sdk-js-v3` · **Version:** `^3.x` · Used by TD-02 (presigned PUT for parts) + TD-07 (presigned GET for stream/download).

Generates time-limited signed URLs so the browser talks to storage directly (never through the API):

```ts
import { getSignedUrl } from "@aws-sdk/s3-request-presigner";
import { S3Client, GetObjectCommand, UploadPartCommand } from "@aws-sdk/client-s3";

// TD-07 — streaming/download GET (S3/MinIO serves HTTP Range/206 natively):
const streamUrl = await getSignedUrl(s3, new GetObjectCommand({ Bucket, Key }), { expiresIn: 3600 });
// download variant: add ResponseContentDisposition: 'attachment; filename="..."' to the command input.

// TD-02 — presigned PUT for each multipart part:
const partUrl = await getSignedUrl(
  s3,
  new UploadPartCommand({ Bucket, Key, UploadId, PartNumber }),
  { expiresIn: 3600 },
);
```

- `getSignedUrl(client, command, { expiresIn })` — `expiresIn` in seconds (default 900). Keep expiries short; the API gates issuance (auth) so access control is preserved without proxying bytes (TD-07).
- Playback: the browser `<video src=streamUrl>` issues Range requests; S3/MinIO answers `206 Partial Content` — no custom range code in the API.
- Alternative for browser POST uploads: `@aws-sdk/s3-presigned-post`'s `createPresignedPost` (url + fields). Not needed for the multipart-PUT strategy chosen in TD-02, noted for completeness.
