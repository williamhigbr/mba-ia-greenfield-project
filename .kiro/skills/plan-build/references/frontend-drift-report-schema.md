# frontend-drift-report.md — Schema Reference

Canonical format definition for `frontend-drift-report.md` files emitted by drift-audit SIs (`SI-NN.X.0`).

This file is the **single source of truth** for the report's shape. Both `/plan-build` (author of the emit rules) and `/implement` (consumer that reads, applies, and re-writes report sections) MUST conform to this schema. Operational logic — when to invoke an audit, how to compose Decision values, how to apply them at SI-Xa — lives in the respective SKILL.md files; this document defines only the file format.

---

## Location

`docs/phases/phase-NN-{slug}/frontend-drift-report.md`

Sibling of the plan document, same folder as `progress.md` and `library-refs.md`. The report is scoped to a single phase; multi-phase aggregation is not supported.

The plan folder lives outside the target subproject root (e.g., `next-frontend/`, `apps/web/`) by construction. Acceptance-criteria checks of the form `git diff --name-only HEAD -- <target-subproject>` therefore exclude the report by path scope.

---

## Frontmatter

```yaml
---
kind: drift-report
phase: phase-NN-{slug}
plan_mtime: "<ISO timestamp from plan doc at first audit-SI run>"
---
```

**Field rules:**

- `kind`: literal string `drift-report` (machine-readable discriminator)
- `phase`: matches the phase folder slug (e.g., `phase-02-auth-frontend`)
- `plan_mtime`: ISO-8601 timestamp captured from the plan doc's mtime at the moment the first audit-SI of the phase ran. Used for staleness detection downstream (e.g., re-audit triggering when the plan has been edited after the audits ran). Tracked once at file creation; not updated on per-screen re-runs.

---

## Body

The body is a hierarchical Markdown document, NOT a table. Each screen's section contains a `Quick scan` summary line, a bullet TOC of components, and one `### {path} — {component-name}` H3 section per component with vertical fields.

```markdown
# phase-NN-{slug} — Drift Report

## Screen: {slug} — audited at SI-NN.X.0 ({YYYY-MM-DD})

**Quick scan:** {a} alinhado · {b} drift menor · {c} drift relevante · {d} ausente

- {ComponentName} (`{path-basename}`) → `{status}` {optional-suffix}
- ... (one bullet per component, in audit order)

### {path} — {ComponentName}

- **Status:** {status enum value}
- **Decision:** `{verb}` {inline payload OR newline + bulleted specifics — see § Field rules}
- **Prior:** {empty marker OR prior summary}

### {next-path} — {NextComponentName}

...

## Screen: {next-slug} — audited at SI-NN.Y.0 ({YYYY-MM-DD})
...
```

**Heading format:**

- File-level H1: `# phase-NN-{slug} — Drift Report` (literal, written once on file creation)
- Per-screen H2: `## Screen: {slug} — audited at SI-NN.X.0 ({YYYY-MM-DD})`
  - `{slug}` is the screen's identifier (kebab-cased form of the UI Contract `#### Screen: {name}` heading; e.g., plan's `#### Screen: Signup` → report's `## Screen: signup`)
  - `SI-NN.X.0` is the audit-SI ID that owns this section
  - `{YYYY-MM-DD}` is the date of the most recent run of this audit-SI (overwritten on re-run; see § Section ownership)
- Per-component H3: `### {path} — {ComponentName}`
  - `{path}` is project-relative (e.g., `next-frontend/components/ui/button.tsx`)
  - `{ComponentName}` is the display name from the screen's Reused DS list

**TOC format (Quick scan + bullet list):**

The Quick scan line is a one-line tally of the 4 status enum values; the bullet list mirrors audit order with one bullet per component. Both are **auto-generated by the audit-SI on every write/overwrite** — they are NOT user-editable; user override targets the H3 sections only (see § File-based user override mechanism).

> _Note: if the user overrides Decision verbs at the file-based checkpoint between the audit-SI and SI-Xa, the TOC is NOT regenerated until the audit-SI is re-run. SI-Xa parses H3 sections directly and ignores the TOC, so behavior remains correct, but the TOC suffixes may visibly drift from H3 state until the next re-run._

Suffix conventions on TOC bullets (computed deterministically from the H3 section's Decision verb, NOT free-form):

| Status | Decision verb | Suffix |
|--|--|--|
| `alinhado` | `skip` | _(none)_ |
| `drift menor` / `drift relevante` | `auto-Edit` (single specifics bullet) | _(none)_ |
| `drift menor` / `drift relevante` | `auto-Edit` (N>1 specifics bullets) | `(N changes)` |
| `drift menor` / `drift relevante` | `exception` | `(exception)` |
| `drift relevante` | `CONFLICT` | `(CONFLICT)` |
| `componente ausente` | `create` | `(create)` |

## Field rules

### `### {path} — {ComponentName}` (H3 heading)

Each component the screen depends on (per its `**Reused DS components:**` UI Contract list) gets exactly one H3 section. The heading carries two pieces of identity:

- **Path** — relative to project root, pointing at the actual DS file location per the project's frontend conventions. The directory layout is project-defined (e.g., `<frontend-subproject>/components/ui/<primitive-name>.<ext>` in shadcn-based projects; layout differs in other DS conventions).
- **Component name** — display name as it appears in the screen's Reused DS list (e.g., `ButtonPrimitive`, `BrandLogo`, `ArrowBackIcon`).

Order within a screen: same order as the screen's Reused DS list in the plan. Audit-SI does not reorder.

### `**Status:**`

Exactly one of 4 enum values (case-sensitive, no abbreviations):

| Status | Discriminator |
|--|--|
| `alinhado` | DS file's value byte-matches Figma demand. Same Tailwind utility OR same custom property reference OR mathematical equivalence (e.g., `text-[14px]` ≡ `text-sm` when `text-sm = 0.875rem = 14px`) |
| `drift menor` | Deviation exists but stays within DS token system, same token family/category. Examples: `rounded-xl` vs `rounded-md` (both radius tokens); `font-medium` vs `font-normal` (both font-weight tokens); `h-1.5` vs `h-1` (both spacing tokens); `border` vs `border-2` (both border-width tokens) |
| `drift relevante` | (a) **Missing surface**: variant, prop, or state demanded but not declared in DS file. (b) **System breakage**: Figma uses one DS token family, our component uses a different family (e.g., `var(--card)` vs `var(--background)`). (c) **Hardcoded leak**: hex/rgb literal in DS instead of token. (d) **Component file absent on disk**. (e) **Different state expression mechanism** between DS and Figma demand |
| `componente ausente` | File listed in screen's Reused DS list does not exist on disk |

The audit iterates only the Reused DS list, so a 5th status describing DS components NOT used by Figma is never produced. Any value outside the 4-value enum is invalid.

### `**Decision:**`

One of 5 valid verbs. The Status determines which subset is permitted. Each Decision is rendered as a top-level bullet (`- **Decision:** \`<verb>\``) optionally followed by **nested specifics bullets** (2-space indent) — except `skip`, which is always inline (no nested bullets). This is the **hybrid intelligent format**: verbs without payload stay inline; verbs with payload always render as a header bullet + nested specifics, even when there's only one specifics item (consistency for parser + user override).

| Verb | Permitted on Status | Body shape | Meaning |
|--|--|--|--|
| `skip` | `alinhado` | inline (no nested bullets) | No action; SI-Xa treats as no-op |
| `auto-Edit` | `drift menor`, `drift relevante` | bulleted (≥1 nested specifics) | SI-Xa applies one Edit on the named DS file per nested bullet. Each bullet is one of the 3 forms in § "Naming constraint" below (exact retune / alias retune / additive). Multi-dimension drift (e.g., `size` + `padding` + `gap` all drifted) renders as multiple bullets, one per dimension |
| `exception` | `drift menor`, `drift relevante` | bulleted (1 nested `reason:` bullet) | Explicit no-action with required justification; SI-Xa treats as no-op |
| `create` | `componente ausente` | bulleted (1 nested `source:` bullet) | SI-Xa creates the file (icon, component); the `source:` bullet describes provenance (e.g., `from Figma asset (asset name/role)`) |
| `CONFLICT` | `drift relevante` only | bulleted (`**Note:**` + `**Verb body:**` sub-section) | Informational classification combined with a concrete sub-decision. The `**Note:**` bullet is human-readable context (audit-owned, NOT user-editable); the `**Verb body:**` sub-section nests its own verb (`auto-Edit` or `exception` only — `create`/`skip` categorically incompatible with `drift relevante`) followed by nested specifics. Default verb body is `auto-Edit` form 3 (additive) per § "Naming constraint" below — retune (forms 1 + 2) is structurally unsafe in CONFLICT cases because it modifies existing variants and could break the prior screen. User may override at the file-based checkpoint |

Canonical shapes by verb (rendered under the H3 of one component):

```markdown
### {path} — {ComponentName}

- **Status:** alinhado
- **Decision:** `skip`
- **Prior:** _(none)_
```

```markdown
### {path} — {ComponentName}

- **Status:** drift relevante
- **Decision:** `auto-Edit`
  - retune size: default per Figma 'md' demand — h-9 → h-10
  - retune padding: default — px-3 → px-4
  - retune gap: default — gap-2 → gap-3
- **Prior:** _(none)_
```

```markdown
### {path} — {ComponentName}

- **Status:** drift menor
- **Decision:** `exception`
  - reason: preserves prior visual identity — divergence is intentional per design lead
- **Prior:** _(none)_
```

```markdown
### {path} — {ComponentName}

- **Status:** componente ausente
- **Decision:** `create`
  - source: from Figma asset (icon: arrow-back)
- **Prior:** _(none)_
```

```markdown
### {path} — {ComponentName}

- **Status:** drift relevante
- **Decision:** `CONFLICT`
  - **Note:** divergent demand vs prior signup screen — current screen demands `size: lg`, prior demanded `size: md`
  - **Verb body:** `auto-Edit`
    - +size 'lg' from Figma demand
- **Prior:** "additive +size 'md' at SI-02.7.0"
```

**Default decisions per status** (emitted by the audit-SI; user can override at the file-based checkpoint between the audit-SI and SI-Xa):

| Status | Default verb | Default body shape | User override |
|--|--|--|--|
| `alinhado` | `skip` | inline | rarely needed |
| `drift menor` | `auto-Edit` | ≥1 specifics bullet, per § Naming constraint | can override verb to `exception` (preserves existing value) — must add `reason:` bullet |
| `drift relevante` | `auto-Edit` | ≥1 specifics bullet, per § Naming constraint — try retune (exact-name OR alias-mapped) before falling back to additive (see /implement SKILL.md sub-step 4 for algorithm). On cross-screen divergence the verb is upgraded to `CONFLICT` with `**Note:**` + `**Verb body:**` sub-section | can override verb to `exception` (with `reason:` bullet); for CONFLICT rows can override the inner `**Verb body:**` to a different verb/specifics; warning: overriding to `exception` or to retune in CONFLICT may break visual fidelity or prior screens |
| `componente ausente` | `create` | 1 `source:` bullet | rarely overridden |

### Naming constraint for `auto-Edit` specifics bullets

Each specifics bullet under `auto-Edit` (and under the `**Verb body:**` of a CONFLICT) carries one drift-resolution string. Variant / size / prop / state names within these strings MUST be byte-derived from the Figma component property values returned by `get_design_context` (the audit-SI's primary input). Inventing names from screen context — screen slug, route name, feature name, business domain (e.g., `'auth'`, `'login'`, `'signup'`, `'checkout'`) — is **forbidden**.

Three valid forms per bullet (consumer skill `/implement` tries them in order; first match wins):

| Form | When | Example bullet text |
|--|--|--|
| `retune <prop>: <existing-variant> — <old> → <new>` | DS file has a variant with the same name as the Figma demand value, but values drift | `retune size: md — h-9 → h-10` |
| `retune <prop>: <code-variant> per Figma '<figma-name>' demand — <old> → <new>` | Project's design-system rule documents an alias from the Figma name to an existing DS variant (e.g., shadcn convention `md ≡ default`); audit retunes the aliased existing slot, preserving the Figma name in the audit trail | `retune size: default per Figma 'md' demand — h-9 → h-10` |
| `+<dim> '<figma-name>' from Figma demand` | Figma demand has a name with no exact match in DS and no alias documented; audit emits truly additive bullet with the literal Figma name | `+variant 'narrow' from Figma demand` |

Multiple drift dimensions on one component → multiple bullets (one per dimension). Each bullet is independently parseable + applyable by SI-Xa. The audit-SI's classification algorithm (consumer skill `/implement` § audit-SI sub-step 4) handles the alias-map lookup by reading the project's design-system rule (typically `.kiro/rules/design-system.md` § "Variant Name Aliases (Figma → code)" — project convention; absence of file or section means alias map is empty and only forms 1 + 3 apply). This schema only constrains the format of emitted bullets; the lookup mechanics live in the consumer.

### Variant-conflict bullets (audit-SI sub-step 4 emit only)

In addition to the 3 Figma-name-anchored forms above, the audit-SI may emit **variant-conflict bullets** that target stale code-side opinions on the SAME property the Figma demand is tuning. These bullets are NOT subject to the Figma-name byte-derivation rule (there is no Figma name to anchor — the bullet is about cleanup of pre-existing utilities, not about adding/retuning to match Figma).

Two valid forms:

| Form | When | Example bullet text |
|--|--|--|
| `drop variant override: <variant>:<prop>-<value> (conflicts with <reason>)` | DS file has a sibling utility `<variant>:<prop>-<value>` (e.g., `dark:bg-input/30`) on the same property the audit is tuning under base form, AND the project has a token (e.g., `--input-background`) explicitly designed to cover the variant's mode via cascade. The variant utility's higher specificity (`0,2,0` for single-variant prefix vs `0,1,0` for base) would override the base under that variant unless dropped | `drop variant override: dark:bg-input/30 (conflicts with token-driven --input-background)` |
| `retune variant override: <variant>:<prop>: <old> → <new>` | DS file has a legitimate state utility (e.g., `hover:bg-muted`) but Figma's resolved value for that state diverges from the existing one | `retune variant override: hover:bg-muted → bg-accent` |

**Discriminator (drop vs retune) — concrete rule:**

- **Drop** when ALL three conditions hold: (1) the audit's primary `auto-Edit` for the same property uses a project-level dual-mode token (a custom property with light + dark resolutions defined in `globals.css` or equivalent — e.g., `--input-background`); (2) the existing `<variant>:` utility's variant prefix scopes the same mode the dual-mode token already covers (e.g., `dark:` ↔ token's dark resolution); (3) the dual-mode token's resolution in that mode differs from (or replaces the role of) the variant utility's value. Together this means: the project chose the token-paradigm to cover the mode; the `<variant>:` opinion is a stale residual from the library scaffold (typical with `npx shadcn add`) that would override the token via specificity. Emit `drop variant override:`.
- **Retune** when the variant utility scopes a state OTHER than the one a dual-mode token covers — typically `hover:`, `focus:`, `focus-visible:`, `aria-invalid:`, `data-[state=...]:`, etc. — and Figma's resolved value for that state diverges from the existing one. The state semantics are kept; only the value flips. Emit `retune variant override:`.

In practice: `dark:` variants paired with a dual-mode token → drop. State variants (`hover:`, `focus:`, `aria-*:`, `data-*:`) → retune (or skip if value matches).

**Project-awareness:** The audit-SI reads `globals.css` (or the project's design-system rule file) before emitting variant-conflict bullets, building a list of dual-mode tokens. When the property under tuning has such a token AND the DS file has a **mode-covering** `<variant>:<prop>-...` opinion (typically `dark:`, scoping the same mode the token already covers via cascade), the bullet IS `drop variant override:`. **State-scoped variants** (`hover:`, `focus:`, `focus-visible:`, `aria-*:`, `data-*:`, etc.) are NOT mode-covering and follow the retune-or-skip path regardless of dual-mode token presence — retune when Figma's same-state demand drifts from the existing value, skip when it matches. The user can override at the file-based checkpoint.

**Application semantics in SI-Xa:** `drop variant override:` → remove the named utility from the class string. `retune variant override:` → replace the named utility's value while keeping its variant prefix. Both are mechanical Edits on the cited DS file.

The post-edit variant-conflict guard in SI-Xa (consumer skill `/implement` § SI-Xa sub-step 2 "Post-edit variant-conflict guard") provides defense-in-depth: if SI-Xa applies a base-form `auto-Edit` AND the file has a sibling variant utility on the same property AND the Drift Report does NOT carry a corresponding `drop variant override:` / `retune variant override:` bullet, SI-Xa halts and asks the user to resolve. This catches drift that the audit's regex enumeration missed.

### `**Prior:**`

One of 4 lifecycle states (rendered as a single inline bullet — Prior never has nested specifics, regardless of complexity):

| Prior format | Meaning |
|--|--|
| `_(none)_` | First-time appearance in this phase; no cross-screen lineage |
| `"<prior decision summary> at SI-NN.X.0 honored"` | Current outcome reproduces prior decision (no divergence); status is `alinhado` because file already reflects the prior `auto-Edit` |
| `"<prior summary> at SI-NN.X.0"` | Current diverges from prior outcome; this component's Decision verb is `CONFLICT` (see § Decision) |
| `"exception at SI-NN.X.0 — '<reason>'"` | Prior decision was `exception` (no auto-Edit was applied; file in original state). The component is reclassified standalone via the discriminator; lineage is informational only, no CONFLICT framing |

The `SI-NN.X.0` reference points at the **source audit-SI** that recorded the prior decision (e.g., `SI-02.7.0` if signup screen recorded the original decision and login screen is now consulting it). The audit-SI ID is the unique cross-screen reference key.

## Section ownership and update discipline

- A given screen's section is **owned** by its audit-SI (`SI-NN.X.0`). Other audit-SIs in the same phase NEVER modify it.
- **First write** (audit-SI's first run) APPENDS the section in execution order — the same order in which audit-SIs appear in the plan (e.g., screen X → screen Y → screen Z for a phase with 3 screens).
- **Re-run** of an audit-SI OVERWRITES its own section in place (idempotent); date in heading reflects last run.
- The "append" discipline applies **across screens**; the "overwrite" discipline applies **within a single screen's re-runs**. Prior sections (other screens) are read-only inputs to the cross-screen consultation algorithm executed by the current audit-SI.

**File creation:** the file is created on the first audit-SI run of the phase (`Created: frontend-drift-report.md.` line in the per-SI completion report). Subsequent audit-SIs append their section without rewriting the frontmatter or H1 (`Updated: frontend-drift-report.md.`).

**Frontmatter immutability:** `plan_mtime` is captured once at file creation and not updated on per-screen re-runs. Staleness detection (re-audit triggering) is a downstream responsibility of `/implement` preflight, not the file format.

## File-based user override mechanism

Between the audit-SI completing and SI-Xa starting, `/implement` pauses with `Seguir para SI-NN.Xa?`. The user opens this file, optionally edits Decisions on per-component H3 sections, and responds "Seguir" — at which point SI-Xa reads the (potentially user-edited) report and applies decisions verbatim.

### Editable fields

- `**Decision:**` verb (the backticked word: `skip` / `auto-Edit` / `exception` / `create` / `CONFLICT`)
- Decision specifics bullets nested under the Decision (add, remove, modify text)
- Inside `CONFLICT`: the `**Verb body:**` verb + its own nested specifics
- Inside `exception`: the `reason:` bullet text
- Inside `create`: the `source:` bullet text

### NOT editable

- `### {path} — {ComponentName}` (H3 heading) — the audit's identification of the component
- `**Status:**` line — the audit's classification (if wrong, re-run audit, don't hand-edit)
- `**Prior:**` line — cross-screen lineage tracked by audit
- Inside `CONFLICT`: the `**Note:**` bullet — audit-owned diagnostic, NOT user opinion
- TOC (`Quick scan` line + bullet list of components) — auto-generated by the audit-SI on every write/overwrite; user edits are clobbered on next re-run
- Screen H2 heading + audit date — audit-SI ownership

### Constraint on Decision verb override

The verb chosen by the user MUST be permitted by the component's Status (per § Decision table "Permitted on Status" column):

| Status | Verbs valid as user override |
|--|--|
| `alinhado` | `skip` only (other verbs incompatible — file matches Figma, no action needed) |
| `drift menor` | `auto-Edit` or `exception` |
| `drift relevante` | `auto-Edit`, `exception`, or `CONFLICT` (CONFLICT only valid if `prior_decisions` originally diverged for this component — user cannot manufacture CONFLICT) |
| `componente ausente` | `create` only (no other verb makes sense — file doesn't exist on disk) |

If the user wants to invert the audit's `drift relevante` to "preserve existing", they switch verb to `exception` + add `reason:` bullet — they CANNOT switch to `skip` (skip means "no drift", which contradicts the Status).

### Common override patterns

1. **"Preserve existing value"** → change `auto-Edit` to `exception` + add `reason:` bullet documenting why divergence is intentional.
2. **"Tune the auto-Edit differently"** → keep `auto-Edit` verb, modify the text of one or more specifics bullets.
3. **"Accept retune risk in CONFLICT"** → in a CONFLICT row, change the `**Verb body:**` verb from default `auto-Edit` form 3 (additive) to form 1/2 (retune existing/aliased) — accepting that the prior screen's visuals may break and would need a separate fix.
4. **"Audit missed a drift dimension"** → if the omitted dimension stays within the same status, add a bullet under `auto-Edit`. If it would change Status (e.g., reveal a `componente ausente` row that was wrongly classified `alinhado`), re-run the audit — don't hand-edit.

### Failure mode

If SI-Xa parsing detects malformed structure (verb not in 5-verb enum, verb incompatible with Status, missing required nested bullet for verb shape, CONFLICT without `**Note:**` or `**Verb body:**`), SI-Xa **halts and asks the user to fix the report** before proceeding. The audit's structure is the contract; SI-Xa never improvises.

**Additional runtime halt — uncovered variant sibling.** Beyond parsing failures, SI-Xa also halts at runtime when its **post-edit variant-conflict guard** finds a sibling variant utility in the DS file that the Drift Report did not enumerate (no matching `drop variant override:` / `retune variant override:` bullet for it). This is a separate halt class with a different fix path: the user resolves ad-hoc (drop, retune, or keep) and SI-Xa records the choice in the per-SI completion report's `Updated existing DS:` annotation — the Drift Report file itself is NOT modified retroactively. See consumer skill `/implement` § SI-Xa sub-step 2 "Post-edit variant-conflict guard" for the full guard semantics.

The override is **file-based**, not prompt-based. No `AskUserQuestion` per component; markdown editing is the override mechanism. This produces a git-diffable artifact of the user's design choices.
